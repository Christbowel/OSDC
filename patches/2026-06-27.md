## GHSA-72r4-9c5j-mj57 · pnpm

2026-06-27 · JavaScript · **HIGH** 7.1 · Pattern: `PATH_TRAVERSAL→FILE_DELETE`

### Root cause

The `pnpm patch-remove` command allowed users to specify patch files that were not strictly located within the configured 'patches' directory. The original implementation only checked the immediate parent directory of the patch file, but did not perform robust validation to ensure the resolved path was a true subdirectory of the intended patches directory, especially when symbolic links or path manipulation (e.g., `../`) were involved. This allowed an attacker to craft a `patchFile` entry in `patchedDependencies` that pointed to an arbitrary file outside the patches directory.

### Impact

An attacker could delete arbitrary files on the system where the `pnpm patch-remove` command was executed, provided they could control the `patchedDependencies` configuration. This could lead to denial of service, data loss, or system instability.

### Fix summary

The patch introduces a new utility function `isSubdirectory` to robustly check if a given path is truly a subdirectory of another, considering symbolic links and path normalization. This function is now used to validate both the configured `patchesDir` and each individual `patchFile` to ensure they are strictly contained within the project's lockfile directory and the designated patches directory, respectively, before any file operations are performed.
