## GHSA-45gf-fjxp-cjpq · meta-ads-mcp

2026-07-17 · Python · **HIGH** 8.3 · Pattern: `SSRF→INTERNAL_ACCESS`

### Root cause

The application's `upload_ad_image` functionality, specifically the `download_image` and `try_multiple_download_methods` functions, fetched images from a user-supplied URL without sufficient validation. This allowed an attacker to specify internal or restricted network addresses, causing the server to make requests on their behalf.

### Impact

An attacker could use the vulnerable server as a proxy to access internal services, private networks, or cloud metadata endpoints (e.g., AWS EC2 metadata service at 169.254.169.254), potentially leading to information disclosure or further compromise of the internal infrastructure.

## GHSA-8wr5-jm2h-8r4f · vllm

2026-07-17 · Python · **HIGH** 7.5 · Pattern: `INTEGER_OVERFLOW→BOUNDARY`

### Root cause

The vulnerability existed because the `recovered_id` could exceed the `vocab_size` during token recovery, especially when all valid entries in the last tile had zero probability. This out-of-bounds index was then used to access memory, leading to a crash.

An attacker could craft a malicious input that causes the vLLM server to crash, leading to a denial of service.

## GHSA-937x-gpqr-72gg · Flask-Reuploaded

2026-07-17 · Python · **HIGH** 7.5 · Pattern: `UNCLASSIFIED`

### Root cause

The vulnerability stemmed from an incomplete fix for a previous issue (CVE-2026-27641). The application performed extension validation before a potential name override, but if a name override occurred, the extension was re-validated without normalizing its case. This allowed an attacker to bypass the denylist by using a mixed-case extension (e.g., '.PhP') that would pass the initial check but then be treated as a forbidden extension by the underlying OS or web server after the override.

### Impact

An attacker could upload files with forbidden extensions, potentially leading to remote code execution if the server is configured to execute scripts based on their actual content or if the attacker can trick the server into processing the file in an unintended way.

The patch addresses the vulnerability by ensuring that the basename of the file, especially after a name override, is converted to lowercase before its extension is re-validated against the denylist. This prevents case-folding asymmetry from being exploited to bypass security checks.

## GHSA-c4gh-rv8h-q9vw · @prompty/core

2026-07-17 · JavaScript · **HIGH** 0.0 · Pattern: `UNSANITIZED_INPUT→COMMAND`

### Root cause

The vulnerability stemmed from the `matter` library's default behavior of parsing JavaScript frontmatter. When processing `.prompty` files, the system would execute JavaScript code embedded in the frontmatter, leading to arbitrary code execution.

### Impact

An attacker could embed malicious JavaScript code within a `.prompty` file's frontmatter, which would then be executed by the application. This allows for arbitrary code execution on the server or client, depending on the execution context.

### Fix summary

The patch explicitly configures the `matter` library to reject JavaScript and `js` frontmatter engines. It introduces a `rejectExecutableFrontmatter` function that throws an error when these engines are encountered, preventing the parsing and execution of potentially malicious JavaScript code.
