## GHSA-73cv-556c-w3g6 · mcp-pinot-server

2026-06-26 · Python · **CRITICAL** 10.0 · Pattern: `UNSANITIZED_INPUT→SQL`

### Root cause

The application allowed unauthenticated users to execute arbitrary SQL queries against the Pinot database. The `oauth_enabled=False` default configuration combined with binding to `0.0.0.0` made the Pinot server publicly accessible without authentication, enabling attackers to send malicious SQL.

### Impact

An attacker could execute arbitrary SQL commands, potentially leading to data exfiltration, modification, or deletion, and could also invoke administrative functions or other tools if the underlying database permissions allowed.

## GHSA-jv46-xfwm-36j7 · relyra

2026-06-26 · Erlang · **CRITICAL** 9.1 · Pattern: `MISSING_VERIFICATION→SIGNATURE`

### Root cause

The Relyra SAML library failed to cryptographically verify the SignatureValue element in SAML responses. While it performed checks like algorithm allowlisting and certificate trust, it did not actually perform the cryptographic signature validation against the signed content, allowing an attacker to forge SAML responses.

### Impact

An attacker could bypass authentication by crafting a malicious SAML response with a valid-looking but unverified signature, gaining unauthorized access to resources or impersonating legitimate users.

### Key diff

## GHSA-4c8j-mgm4-qqvp · github.com/umputun/remark42

2026-06-26 · Go · **HIGH** 8.2 · Pattern: `UNCLASSIFIED`

### Root cause

The application allowed user-uploaded images to be served without the X-Content-Type-Options: nosniff header. This omission enabled browsers to perform MIME-sniffing, potentially reinterpreting a maliciously crafted image file (e.g., a GIF or JPG containing HTML/JavaScript) as an executable HTML or JavaScript document, leading to XSS.

### Impact

An attacker could upload a specially crafted 'image' file that, when accessed, would be executed as HTML or JavaScript in the victim's browser, allowing for arbitrary client-side code execution, session hijacking, or defacement.

### Root cause

The application previously used `filter_var` with `FILTER_VALIDATE_URL` to determine if an option value was a URL. This validation was insufficient as it allowed schemes like `file://`, `php://`, and `ftp://` to be processed, leading to the server fetching arbitrary local files or resources from internal networks.

### Impact

An attacker could supply a specially crafted URL to an option (e.g., `attachment`) to read local files from the server (e.g., `/etc/passwd`) or make requests to internal network resources, potentially leading to further compromise.

## GHSA-396q-4vc8-28x9 · @microsoft/kiota-http-fetchlibrary

2026-06-26 · JavaScript · **MODERATE** 0.0 · Pattern: `INFO_DISCLOSURE→ERROR_MESSAGE`

### Root cause

The vulnerability existed because the `redirectHandlerOptions` middleware, responsible for scrubbing sensitive headers like Authorization and Cookie during cross-origin redirects, failed to perform case-insensitive matching for header keys. The `FetchRequestAdapter` lower-cased all header keys before they reached this middleware, making `delete headers.Authorization` or `delete headers.Cookie` ineffective if the original header keys were in PascalCase or any other casing.

### Impact

An attacker could potentially leak sensitive information, such as Bearer tokens or session cookies, to an unintended third-party origin if a redirect occurred. This could lead to unauthorized access to user accounts or services.

### Key diff

```
- delete headers.Authorization;
- delete headers.Cookie;
+ for (const key of Object.keys(headers)) {
+ 	const lower = key.toLowerCase();
+ 	if (lower === "authorization" || lower === "cookie" || lower === "proxy-authorization") {
+ 		delete headers[key];
+ 	}
+ }
```

### Fix summary

The patch modifies the `redirectHandlerOptions` middleware to iterate through all header keys, convert them to lowercase, and then check for 'authorization', 'cookie', and 'proxy-authorization'. If a match is found, the header is deleted, ensuring that sensitive headers are properly scrubbed regardless of their original casing.

The `hackney` HTTP client library, when configured with a per-chunk timeout, allowed an attacker to send a large HTTP body very slowly, one byte at a time. This caused the server to accumulate the entire body in memory without a total body timeout, leading to unbounded memory consumption.

An attacker could exhaust the server's memory, causing it to crash or become unresponsive, leading to a denial of service for legitimate users.


### Root cause
