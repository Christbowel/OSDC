## GHSA-2gr4-ppc7-7mhx · codeigniter4/framework

2026-06-11 · PHP · **CRITICAL** 9.8 · Pattern: `UNCLASSIFIED`

### Root cause

The vulnerability existed because the `ext_in` validation rule only checked the guessed file extension, which could be manipulated by an attacker. The `guessExtension()` method might return an empty string or an incorrect extension if the file's MIME type or content was malformed, allowing a malicious file with a dangerous extension (e.g., .php) to bypass the intended extension whitelist.

### Impact

An attacker could upload files with disallowed extensions, potentially leading to remote code execution if the server is configured to execute scripts based on their extension, or other forms of system compromise.

### Key diff

```
-            if (! in_array($file->guessExtension(), $params, true)) {
+            $clientExtension = strtolower($file->getClientExtension());
+
+            if ($clientExtension === '' || ! in_array($clientExtension, $params, true)) {
+                return false;
+            }
+
+            if ($file->guessExtension() !== $clientExtension) {
                 return false;
             }
```

### Fix summary

The patch enhances the `ext_in` validation rule by explicitly checking both the client-provided file extension (`getClientExtension()`) and comparing it with the guessed extension (`guessExtension()`). It ensures that the client extension is not empty and is part of the allowed list, and that the guessed extension matches the client extension, preventing bypasses through manipulated file types.

## GHSA-4x76-22x2-rx8v · @openzeppelin/wizard

2026-06-11 · JavaScript · **HIGH** 8.8 · Pattern: `UNSANITIZED_INPUT→COMMAND`

### Root cause

The OpenZeppelin Contracts Wizard generated Hardhat and Foundry tests by directly embedding user-supplied `opts.name` and `opts.uri` values into string literals without proper escaping. This allowed an attacker to inject arbitrary code into the generated test files by crafting malicious input that would break out of the string literal.

### Impact

An attacker could inject arbitrary code into the generated test files, potentially leading to remote code execution if these tests were run in a compromised environment, or could manipulate the test logic to pass malicious contracts.

### Key diff

```
--- a/packages/core/solidity/src/zip-foundry.ts
+++ b/packages/core/solidity/src/zip-foundry.ts
-          return ['function testName() public view {', [`assertEq(instance.name(), "${opts.name}");`], '}'];
+          return ['function testName() public view {', [`assertEq(instance.name(), ${stringifyUnicodeSafe(opts.name)});`], '}'];
--- a/packages/core/solidity/src/zip-hardhat.ts
+++ b/packages/core/solidity/src/zip-hardhat.ts
-          return [`expect(await instance.name()).to.equal("${opts.name}");`];
+          return [`expect(await instance.name()).to.equal(${JSON.stringify(opts.name)});`];
```

### Fix summary

The patch introduces and utilizes `stringifyUnicodeSafe` for Foundry tests and `JSON.stringify` for Hardhat tests to properly escape user-controlled input (`opts.name` and `opts.uri`) before embedding it into the generated test code. This ensures that the input is treated as a literal string and cannot inject code.

## GHSA-wxq4-cc2q-338q · wsgidav

2026-06-11 · Python · **HIGH** 7.1 · Pattern: `PATH_TRAVERSAL→FILE_READ`

### Root cause

The vulnerability existed because the path containment check was not robust enough. It only checked if the `file_path` started with `root_path`, which could be bypassed if a directory name started with the root path but was not actually within it (e.g., `/tmp/share_evil` vs `/tmp/share`). This allowed specially crafted URLs with encoded dot segments to escape the intended share root.

### Impact

An attacker could access, read, write, or delete files outside the configured WebDAV share root, potentially leading to sensitive information disclosure, unauthorized data modification, or even remote code execution if critical system files could be overwritten.

### Key diff

```

+++ b/wsgidav/fs_dav_provider.py
@@ -457,9 +457,14 @@ def _loc_to_file_path(self, path: str, environ: dict = None):

-        if not file_path.startswith(root_path) and not is_shadow:
+        # Ensure the containment check is path-boundary-aware: append os.sep so
```

### Fix summary

The patch strengthens the path containment check by appending a directory separator (`os.sep`) to both the `root_path` and `file_path` before performing the `startswith` check. This ensures that the comparison is boundary-aware, preventing partial matches that could lead to path traversal.

## GHSA-9663-mqmp-p9mm · zeroconf

2026-06-11 · Python · **MODERATE** 6.5 · Pattern: `DOS→RESOURCE_EXHAUSTION`

### Root cause

The zeroconf library's mDNS listener had an unbounded queue for reassembling fragmented (TC-bit set) DNS packets. An attacker could exploit this by sending a flood of spoofed-source fragmented packets, causing the server to allocate an ever-increasing amount of memory to store the incomplete fragments, leading to memory exhaustion.

### Impact

An attacker on the local network could exhaust the memory of a system running the vulnerable zeroconf service, leading to a denial of service for the application or the entire system.

### Key diff

### Fix summary

The patch introduces limits on the number of distinct IP addresses for which deferred packets are stored (_MAX_DEFERRED_ADDRS) and the number of packets stored per IP address (_MAX_DEFERRED_PER_ADDR). It also adds logic to evict the oldest deferred address's state when the total limit is reached, preventing unbounded memory growth.

## GHSA-x426-x7cc-3fpc · @hapi/wreck

2026-06-11 · JavaScript · **MODERATE** 6.5 · Pattern: `SSRF→INTERNAL_ACCESS`

### Root cause

The @hapi/wreck library, when handling HTTP redirects, only removed sensitive credential headers (like Authorization or Cookie) if the hostname changed. It failed to consider changes in scheme (HTTP to HTTPS) or port, leading to these headers being sent to potentially untrusted origins on the same host or different ports.

### Impact

An attacker could potentially redirect a user to a malicious server on a different port or scheme on the same host, or a different host entirely, and capture sensitive credential headers, leading to session hijacking or unauthorized access.

### Key diff

```
-                if (uri.hostname !== parsedLocation.hostname) {
+                if (uri.origin !== parsedLocation.origin) {
```

### Fix summary

The patch modifies the redirect logic to remove sensitive credential headers not only when the hostname changes, but also when the scheme or port changes. This ensures that credentials are not leaked across any cross-origin redirects.
