## GHSA-mw3q-r9wh-h2ff · nimiq-primitives

2026-05-21 · RUST · **HIGH** 7.5 · Pattern: `DOS→RESOURCE_EXHAUSTION`

### Root cause

The vulnerability stemmed from the `MerkleRadixTrie` implementation not properly validating input `TrieChunk` items. Specifically, it allowed an item with an empty (ROOT) key to be processed, which would then cause a panic in the `put_value` method when attempting to store a value at the root node, as the root node is not designed to hold values.

### Impact

An attacker could craft a malicious `TrieChunk` containing a ROOT-keyed item and send it to a node, causing the node to panic and crash. This leads to a denial of service for the affected node.

## GHSA-799f-29jm-gr6c · nimiq-primitives

2026-05-21 · RUST · **MODERATE** 5.9 · Pattern: `UNCLASSIFIED`

### Root cause

The `BlockInclusionProof::verify` method incorrectly returned `true` when the `hops` array was empty, without verifying if the `election_head` block actually referenced the `target` block. This allowed an attacker to claim a block was included even if it wasn't directly referenced.

### Impact

An attacker could potentially forge block inclusion proofs, leading to incorrect state synchronization or validation within the blockchain network, potentially enabling double-spending or other consensus-breaking attacks.

### Fix summary

The patch modifies the `BlockInclusionProof::verify` method. When `hops` are empty, it now explicitly calls a new helper function `references_block` to check if the `election_head` directly references the `target` block via its `parent_election_hash` or `interlink` before returning `true`.

## GHSA-vghx-352f-93jm · nimiq-blockchain

2026-05-21 · RUST · **MODERATE** 5.3 · Pattern: `UNCLASSIFIED`

### Root cause

The vulnerability stemmed from an unhandled error case when requesting a batch set for the genesis block. Specifically, the `prove_num_leaves` function, which calculates history length, would panic if called for the genesis block because the genesis block does not have a preceding macro block, leading to an unrecoverable error.

An attacker could send a crafted `RequestBatchSet` message targeting the genesis block, causing the node to panic and crash, leading to a denial of service for that node.

### Fix summary

The patch introduces a new error type `CouldntProduceHistorySizeProof` and wraps the `prove_num_leaves` calls in a `Result` to handle potential errors gracefully. 2026-05-21 · RUST · **MODERATE** 4.4 · Pattern: `PATH_TRAVERSAL→FILE_READ`

The application constructed file paths by directly appending user-controlled section entry names to a base directory path without proper sanitization or validation. This allowed special path components like '..' or absolute paths to escape the intended directory.

### Impact

## GHSA-h9cc-w26m-j342 · nimiq-keys

2026-05-21 · RUST · **MODERATE** 4.3 · Pattern: `DOS→RESOURCE_EXHAUSTION`

### Root cause

The vulnerability stemmed from the `delinearize` function in the Ed25519 public key operations. Previously, this function used `unwrap()` on the result of `to_edwards_point()`, assuming that a valid public key would always convert to a valid Edwards point. However, if an attacker provided a public key that was not a valid curve point, this `unwrap()` call would panic, leading to a denial of service.

### Impact

An attacker could provide a malformed public key, causing the application to panic and crash when attempting to delinearize it. This would lead to a denial of service for any operation involving the processing of such a public key, such as multisig commitment building or key aggregation.
