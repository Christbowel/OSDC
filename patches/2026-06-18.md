## GHSA-xqxv-4jc2-x56x · github.com/zitadel/zitadel

2026-06-18 · Go · **HIGH** 7.4 · Pattern: `MISSING_VERIFICATION→SIGNATURE`

### Root cause

The vulnerability stemmed from a violation of RFC 6749 Section 4.1.3, specifically the lack of client_id binding in OIDC authorization code exchange and refresh token flows. The system did not verify that the client_id provided in the token request matched the client_id associated with the authorization code or refresh token, allowing a malicious client to use a valid authorization code or refresh token issued to another client.

### Impact

An attacker could potentially impersonate another client application, gain unauthorized access to resources, or elevate privileges by exchanging a valid authorization code or refresh token (obtained for a different client) for an access token.

## GHSA-jr45-52cw-69h5 · nl.nl-portal:documenten-api

2026-06-18 · Java · **MODERATE** 6.5 · Pattern: `MISSING_AUTHZ→RESOURCE`

### Root cause

The application exposed a GraphQL endpoint `getDocumentContent` that allowed any authenticated user to download the content of any document by providing its ID and API. This endpoint lacked proper authorization checks to ensure the requesting user had permissions for the specific document, leading to an Insecure Direct Object Reference (IDOR) vulnerability.

### Impact

An attacker, once logged in, could download the content of any document stored in the system, regardless of whether they were authorized to access that specific document. This could lead to sensitive data exposure.

## GHSA-q6r4-3wmg-fwcq · github.com/containers/podman/v3

2026-06-18 · Go · **MODERATE** 5.3 · Pattern: `PATH_TRAVERSAL→FILE_READ`

### Root cause

The vulnerability stemmed from an insecure resolution of the container's working directory (WORKDIR). Podman's internal logic for resolving WORKDIR did not properly handle symbolic links, especially when they pointed outside the intended container mountpoint. This allowed an attacker to craft a malicious image with a WORKDIR that, when resolved, could traverse outside the container's root.

### Impact

An attacker could potentially escape the container's intended filesystem boundaries, leading to arbitrary file access (read/write) on the host system or other mounted volumes, depending on the context in which Podman was running.

### Key diff

```
--- a/libpod/container_internal_common.go
+++ b/libpod/container_internal_common.go
@@ -904,7 +859,7 @@ func (c *Container) resolveWorkDir() error {
 		return nil
 	}
 
-	_, resolvedWorkdir, _, err := c.resolvePath(c.state.Mountpoint, workdir)
+	resolvedWorkdir, err := securejoin.SecureJoin(c.state.Mountpoint, workdir)
```
