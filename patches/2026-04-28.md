## GHSA-35hp-hqmv-8qg8 · github.com/gofiber/fiber/v3

2026-04-28 · Go · **MODERATE** 6.5 · Pattern: `INSECURE_DEFAULT→CONFIG`

### Root cause

The Fiber cache middleware's default key generator did not properly account for query string parameters. Specifically, in certain scenarios (e.g., when query string length exceeded a buffer, or when too many parameters were present), it would use the raw, unescaped query string as part of the cache key without proper canonicalization or escaping. This led to different URLs with distinct query parameters being mapped to the same cache key.

### Impact

An attacker could craft requests with different query parameters that would be served the same cached response. This could lead to information disclosure (e.g., a user seeing another user's cached data) or response manipulation, where an attacker could cache a malicious response for a legitimate request.

### Key diff

```
--- a/middleware/cache/cache.go
+++ b/middleware/cache/cache.go
@@ -1332,12 +1332,12 @@ func canonicalQueryString(uri *fasthttp.URI) string {
 	// Pre-scan query string to detect excessive parameters before expensive parsing.
 	// This prevents DoS via url.ParseQuery allocating large maps/slices.
 	if len(query) > maxQueryBufferSize {
-		return boundKeySegment(query)
+		return boundKeySegment(escapeKeyDelimiters(query))
 	}
 
 	// Fast path: single key=value pair needs no parsing or sorting
 	if strings.IndexByte(query, '&') < 0 {
-		return boundKeySegment(query)
+		return boundKeySegment(escapeKeyDelimiters(query))
 	}
```

### Fix summary

The patch modifies the `canonicalQueryString` function to ensure that the query string segments used for cache key generation are always escaped using `escapeKeyDelimiters` before being passed to `boundKeySegment`. This ensures that query parameters are properly handled and distinct query strings result in distinct cache keys. Additionally, the method normalization logic was updated to avoid mutating the caller's slice directly.

## GHSA-88hf-wf7h-7w4m · OpenTelemetry.Exporter.Zipkin

2026-04-28 · C# · **MODERATE** 5.3 · Pattern: `DOS→RESOURCE_EXHAUSTION`

### Root cause

The application used an unbounded ConcurrentDictionary to cache Zipkin remote endpoints. Over time, as new and unique remote endpoint service names were encountered, this cache would continuously grow without any eviction mechanism, leading to increased memory consumption.

### Impact

An attacker, or even normal application usage with a high cardinality of remote service names, could cause the application to consume excessive memory, potentially leading to an OutOfMemoryException and application crash, effectively denying service.

### Key diff

```
-	private static readonly ConcurrentDictionary<(string, int), ZipkinEndpoint> RemoteEndpointCache = new();
+	private static readonly ZipkinEndpointLruCache RemoteEndpointCache = new(MaxRemoteEndpointCacheSize);
```

### Fix summary

The fix replaces the unbounded ConcurrentDictionary with a new custom LRU (Least Recently Used) cache implementation, `ZipkinEndpointLruCache`, which has a defined maximum capacity (1024 entries). This ensures that the cache size is bounded and old entries are evicted when the capacity is reached.
