## GHSA-58f6-6rj2-3v8r · Steeltoe.Management.EndpointCore

2026-07-02 · C# · **HIGH** 8.2 · Pattern: `MISSING_AUTH→ENDPOINT`

### Root cause

The application used the `Host` header from the HTTP request to determine the port for management endpoint access control. An attacker could spoof the `Host` header to make the application believe the request was coming from the management port, even if it was not.

### Impact

An attacker could bypass the management port isolation and access sensitive management endpoints that should only be available on a dedicated port, potentially leading to information disclosure or further compromise.

### Key diff


### Fix summary

The patch changes the port check from using `context.Request.Host.Port` (which can be spoofed via the Host header) to `context.Connection.LocalPort` (which reflects the actual local port the server is listening on). This ensures that the port check is based on a reliable, server-side value.

## GHSA-794g-x443-36f7 · org.keycloak:keycloak-services

2026-07-02 · Java · **HIGH** 7.7 · Pattern: `MISSING_VERIFICATION→SIGNATURE`

### Root cause

The vulnerability stemmed from improper validation of encrypted SAML assertions. Keycloak failed to correctly verify the signature of the entire SAML message when an assertion was encrypted, allowing an attacker to craft a SAML response where only the assertion was encrypted and signed, but the overall response was not, leading to an authentication bypass.

### Impact

An attacker could bypass authentication and gain unauthorized access to a Keycloak-protected application by manipulating SAML responses, potentially impersonating legitimate users.

## GHSA-3w32-23wj-rxg3 · craftcms/cms

2026-07-02 · PHP · **HIGH** 0.0 · Pattern: `MISSING_AUTHZ→RESOURCE`

### Root cause

The vulnerability stemmed from insufficient permission checks during the 'move folder' operation. ## GHSA-gg9x-qcx2-xmrh · joserfc

2026-07-02 · Python · **HIGH** 0.0 · Pattern: `MISSING_VERIFICATION→SIGNATURE`

### Root cause

The joserfc library's HMAC-based JWS algorithms (HS256, HS384, HS512) did not explicitly check for empty or nil keys during signing and verification. An empty HMAC key results in a deterministic digest that any party can reproduce, effectively making the signature forgeable.

### Impact

An attacker could forge signatures for JWS tokens that were supposedly protected with HMAC algorithms if an empty key was used, leading to unauthorized access or manipulation of data.

### Key diff

### Fix summary

### Root cause

The `SQLChatAgent` in Langroid used a blocklist to prevent dangerous SQL queries, but this blocklist was incomplete. Specifically, it missed several PostgreSQL functions related to file system access (e.g., `pg_read_file`) and other database-specific primitives that could lead to arbitrary file reads or command execution.

### Impact

An attacker could craft malicious SQL queries that, when executed by the `SQLChatAgent`, would allow them to read arbitrary files from the server's file system, execute shell commands, or potentially gain further control over the database server.

## GHSA-qq2c-2q8j-jh27 · craftcms/cms

2026-07-02 · PHP · **HIGH** 0.0 · Pattern: `MISSING_AUTHZ→RESOURCE`

### Root cause

The vulnerability existed because the authorization check for editing an entry was performed before the entry model was fully populated with POST data. This allowed an attacker to modify the author ID (or other fields) in the POST data after the initial authorization check, effectively spoofing authorship or making unauthorized changes.

### Impact

An attacker could spoof the author of an entry, attributing content to another user without their consent. Depending on other permissions, this could also lead to unauthorized modifications of entry properties.

### Key diff

```
--- a/src/controllers/EntriesController.php
+++ b/src/controllers/EntriesController.php
@@ -295,6 +295,8 @@ public function actionSaveEntry(bool $duplicate = false): ?Response
         // Populate the entry with post data
         $this->_populateEntryModel($entry);
 
+        $this->enforceEditEntryPermissions($entry, $duplicate);
+
         if ($forceDisabled) {
             $entry->enabled = false;
         }
```

### Fix summary

The patch moves the `enforceEditEntryPermissions` call to after the entry model has been populated with data from the POST request. This ensures that the authorization check is performed on the entry's final state, including any changes submitted by the user, preventing the pre-check/post-mutation authorization gap.

## GHSA-x5m4-g2cq-52pq · craftcms/cms

2026-07-02 · PHP · **HIGH** 0.0 · Pattern: `MISSING_AUTHZ→RESOURCE`

### Root cause

The original code only checked permissions on a single asset, `assetToReplace ?: $sourceAsset`, which means if one of these was null, the permission check might not have been applied to the other relevant asset. This could allow an attacker to bypass authorization checks when replacing files if only one of the assets was considered for permission validation.

### Impact

An attacker could potentially replace files on assets for which they do not have explicit permissions, leading to unauthorized modification of content or potentially other system files if the asset replacement mechanism allows for path manipulation.
