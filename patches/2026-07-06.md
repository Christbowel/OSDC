## GHSA-239g-2685-54x3 · uu_install

2026-07-06 · RUST · **MODERATE** 6.3 · Pattern: `RACE_CONDITION→TOCTOU`

### Root cause

The `install` utility performed an `unlink-then-create` operation without proper atomic protection. Specifically, it would remove an existing file at the destination path and then attempt to create a new one. In a multi-threaded or multi-process environment, an attacker could create a symbolic link pointing to an arbitrary file in the brief window between the `remove_file` and `create` operations, leading to a Time-of-Check to Time-of-Use (TOCTOU) race condition.

### Impact

An attacker could exploit this race condition to overwrite arbitrary files on the system with the contents of the source file being installed, potentially leading to privilege escalation or denial of service.

## GHSA-mj6p-44ch-cq69 · uu_mkdir

2026-07-06 · RUST · **LOW** 3.3 · Pattern: `RACE_CONDITION→TOCTOU`

### Root cause

The `mkdir` utility on Unix systems, when creating a directory with specific permissions using the `-m` flag, would first create the directory with permissions dictated by the user's umask. Only after creation would it attempt to `chmod` the directory to the desired permissions. This created a time-of-check-to-time-of-use (TOCTOU) race window.

### Impact

During the brief period between directory creation and the `chmod` call, the directory would exist with potentially more permissive permissions than intended (e.g., world-writable if umask allowed). An attacker could exploit this window to access or modify the directory before its permissions were correctly set, potentially leading to information disclosure or unauthorized modification.

### Fix summary

The patch introduces a `UmaskGuard` RAII (Resource Acquisition Is Initialization) mechanism to temporarily set the umask to 0 before calling `mkdir(2)` on Unix systems.
