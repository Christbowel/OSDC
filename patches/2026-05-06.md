## GHSA-rwm7-x88c-3g2p · io.netty:netty-transport-native-epoll

2026-05-06 · Java · **HIGH** 7.5 · Pattern: `DOS→INFINITE_LOOP`

### Root cause

The Netty epoll transport layer did not correctly handle TCP half-closed connections when a RST packet was received. Specifically, if a channel was registered with epoll and then became half-closed, a subsequent RST could lead to the file descriptor (fd) remaining registered with epoll without any events of interest. This caused epoll_wait to continuously report the fd as ready, leading to an infinite loop and CPU exhaustion.

### Impact

An attacker could send a specially crafted RST packet on a half-closed TCP connection, causing the Netty server to enter an infinite loop, consuming 100% CPU, and effectively leading to a denial of service for the application.
