## GHSA-h4ph-crvj-9h92 · pimcore/admin-ui-classic-bundle

2026-05-27 · PHP · **HIGH** 8.8 · Pattern: `DESERIALIZATION→RCE`

### Root cause

The application was using `unserialize()` on user-controlled or attacker-influenced data read from a configuration file without restricting the allowed classes. This allowed an attacker to inject arbitrary PHP objects into the application's scope.

### Impact

An attacker could achieve remote code execution (RCE) by crafting a malicious serialized object that, when deserialized, would trigger dangerous magic methods or object instantiation leading to arbitrary code execution.

### Key diff

```
-	                $dashboards = Serialize::unserialize(file_get_contents($this->getConfigFile()));
-	                if (!empty($dashboards)) {
-	                    $this->dashboards = $dashboards;
+	                $dashboardFile = file_get_contents($this->getConfigFile());
+	                if ($dashboardFile !== false) {
+	                    $dashboards = unserialize($dashboardFile, ['allowed_classes' => false]);
+	                    if (is_array($dashboards) && !empty($dashboards)) {
+	                        $this->dashboards = $dashboards;
```

### Fix summary

The patch adds a check to ensure the file content is not false before attempting to unserialize it. More importantly, it restricts the `unserialize()` function to only allow arrays by setting `allowed_classes` to `false`, preventing the instantiation of arbitrary objects.

## GHSA-36hh-x5p5-jgc8 · @hapi/content

2026-05-27 · JavaScript · **HIGH** 0.0 · Pattern: `UNSANITIZED_INPUT→HEADER`

### Root cause

The `@hapi/content` header parser did not properly handle duplicate parameters in `Content-Type` and `Content-Disposition` headers. This allowed an attacker to provide multiple values for the same parameter, with the parser only considering the first or last, leading to a bypass of upload filters or other security checks that relied on a single, unambiguous parameter value.

### Impact

An attacker could bypass upload filters by smuggling malicious content past security checks. For example, they could specify a 'filename' parameter twice, with one value being safe and the other malicious, causing the application to process the malicious file.

### Key diff

```

+++ b/lib/index.js
@@ -45,6 +45,10 @@ exports.type = function (header) {

+                throw Boom.badRequest('Invalid content-type header: duplicate parameter');
+            }
         }
     }
```

### Fix summary

The patch introduces checks to detect and reject duplicate parameters in `Content-Type` (specifically for `charset` and `boundary`) and `Content-Disposition` headers. It also refines the `Content-Type` regex to be more strict and improves the parsing of extended `Content-Disposition` values to correctly handle `ext-value` overriding regular values while still preventing other duplicate parameter issues.

## GHSA-ph9p-34f9-6g65 · tmp

2026-05-27 · JavaScript · **HIGH** 0.0 · Pattern: `PATH_TRAVERSAL→FILE_WRITE`

### Root cause

The `tmp` package allowed unsanitized user-controlled input in `prefix` and `postfix` options to be concatenated into file paths. This lack of validation meant that path traversal sequences like `../` could be injected, allowing files to be created outside the intended temporary directory.

### Impact

An attacker could create files in arbitrary locations on the file system, potentially leading to denial of service, privilege escalation, or other system compromises depending on the application's context and permissions.

## GHSA-vhjm-w67q-g75c · @hapi/wreck

2026-05-27 · JavaScript · **MODERATE** 0.0 · Pattern: `INFO_DISCLOSURE→ERROR_MESSAGE`

### Root cause

The @hapi/wreck HTTP client library did not properly clear all sensitive headers, specifically 'Proxy-Authorization', when performing cross-hostname redirects. While 'Authorization' and 'Cookie' headers were removed, 'Proxy-Authorization' was overlooked, leading to its unintended transmission to external hosts.

### Impact

An attacker could potentially intercept the 'Proxy-Authorization' header, which might contain credentials, if the client was configured to use a proxy and then redirected to a malicious or compromised external host. This could lead to unauthorized access to the proxy or other systems.

### Key diff

### Fix summary

The patch introduces a new Set `sensitiveCrossHostHeaders` which explicitly lists 'authorization', 'cookie', and 'proxy-authorization'. During cross-hostname redirects, the code now iterates through this set to remove all specified sensitive headers from the redirect options, ensuring 'Proxy-Authorization' is also cleared.
