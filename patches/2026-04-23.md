## GHSA-v529-vhwc-wfc5 · openc3

2026-04-23 · Ruby · **CRITICAL** 9.6 · Pattern: `UNSANITIZED_INPUT→SQL`

### Root cause

The application directly embedded user-controlled input (start_time, end_time, col_name) into SQL queries without proper sanitization or parameterization. This allowed an attacker to inject arbitrary SQL code by crafting malicious input values.

### Impact

An attacker could execute arbitrary SQL commands on the QuestDB time-series database, potentially leading to data exfiltration, modification, or deletion, and could even achieve remote code execution in some database configurations.

### Key diff

```
Before:
query += "WHERE T0.PACKET_TIMESECONDS < '#{start_time}' LIMIT -1"
result = @@conn.exec(query)

After:
query += "WHERE T0.PACKET_TIMESECONDS < $1 LIMIT -1"
query_params << start_time
result = @@conn.exec_params(query, query_params)
```

### Fix summary

The patch modifies the `tsdb_lookup` and `create_table` methods in both Ruby and Python implementations to use parameterized queries. Instead of directly interpolating user input into the SQL string, placeholders ($1, $2 or %s) are used, and the values are passed as separate parameters to the database driver's `exec_params` or `execute` method.

## GHSA-m8mh-x359-vm8m · org.apktool:apktool-lib

2026-04-23 · Java · **HIGH** 7.1 · Pattern: `PATH_TRAVERSAL→FILE_WRITE`

### Root cause

The application constructed file paths for output by concatenating user-controlled input (entry.getTypeName(), entry.getConfig().getQualifiers(), entry.getName(), and ext) without sufficient sanitization. Although there was a `detectPossibleDirectoryTraversal` check, it was bypassed, allowing an attacker to inject path traversal sequences like `../` into the generated `outResPath`.

### Impact

An attacker could craft a malicious APK file that, when processed by Apktool, would write arbitrary files to arbitrary locations on the file system where Apktool is executed, potentially leading to remote code execution or system compromise.

@@ -100,13 +100,8 @@ public void decode(ResEntry entry, Directory inDir, Directory outDir, Map<String
         }

Instead, it directly constructs the `outResPath` without the previous, ineffective check, implying that other, more robust sanitization or path resolution mechanisms are expected to handle this, or that the input sources are now trusted/sanitized upstream.

## GHSA-88gm-j2wx-58h6 · @astrojs/cloudflare

2026-04-23 · JavaScript · **LOW** 2.2 · Pattern: `SSRF→INTERNAL_ACCESS`

### Root cause

The application's image-binding-transform endpoint fetched remote images without explicitly controlling redirect behavior. This allowed an attacker to provide a URL that would redirect to an internal resource, effectively bypassing previous SSRF mitigations.

### Impact

An attacker could potentially access internal network resources or services that were not intended to be publicly exposed, by chaining redirects from an external URL to an internal one.

### Key diff

```
-	const content = await (isRemotePath(href) ? fetch(imageSrc) : assets.fetch(imageSrc));
+	const content = await (isRemotePath(href)
+		? fetch(imageSrc, { redirect: 'manual' })
+		: assets.fetch(imageSrc));
+
+	if (content.status >= 300 && content.status < 400) {
+		return new Response('Not Found', { status: 404 });
+	}
```

### Fix summary

The patch explicitly sets the `redirect` option to 'manual' for `fetch` calls to remote image sources. Additionally, it introduces a check to return a 404 'Not Found' response if the fetched content's status code indicates a redirect (3xx range), preventing the application from following it.
