## GHSA-c6mh-fpjc-4pr3 · yt-dlp

2026-06-16 · Python · **HIGH** 8.3 · Pattern: `UNSANITIZED_INPUT→COMMAND`

### Root cause

The `yt-dlp` utility allowed the creation of certain dangerous file types (like `.desktop`, `.url`, `.webloc`) by not sufficiently sanitizing or restricting the file extensions used for internet shortcut files. An attacker could craft a malicious filename that, when downloaded, would result in a file with an executable extension, bypassing previous sanitization efforts.

### Impact

An attacker could trick a user into downloading a specially crafted file that, when opened, could execute arbitrary commands or redirect the user to malicious websites, leading to potential system compromise or phishing attacks.

2026-06-16 · Python · **HIGH** 8.3 · Pattern: `UNSANITIZED_INPUT→COMMAND`

## GHSA-q8gq-377p-jq3r · vllm

2026-06-16 · Python · **HIGH** 7.5 · Pattern: `DESERIALIZATION→RCE`

### Root cause

The vulnerability stemmed from the use of `assert` statements for security checks, specifically in the `get_act_fn` function. When Python is run with optimizations (e.g., `python -O`), `assert` statements are stripped out, effectively disabling the security check that restricted activation function loading to `torch.nn.modules.`. This allowed an attacker to specify an arbitrary module path.

### Impact

An attacker could achieve arbitrary code execution by supplying a specially crafted activation function name that points to a malicious module or function outside of the intended `torch.nn.modules.` namespace.

### Root cause

The previous `isSameOriginRequest` implementation incorrectly assumed that if `Sec-Fetch-Site`, `Origin`, and `Referer` headers were all absent, the request was safe. An attacker could craft a request from a non-trustworthy origin, using specific browser features (e.g., non-CORS script, `Referrer-Policy: no-referrer`), to omit these headers, thereby bypassing the same-origin check for non-loopback dev server binds.

### Impact

An attacker could bypass the same-origin policy, potentially allowing cross-site requests to the Nuxt dev server from an arbitrary origin, leading to unauthorized access to development server resources or information disclosure.

### Fix summary

Instead of always allowing such requests, it now only permits them if the dev server is bound to a loopback host (e.g., localhost).
