## GHSA-q8gq-377p-jq3r · vllm

2026-06-16 · Python · **HIGH** 7.5 · Pattern: `DESERIALIZATION→RCE`

### Root cause

The vulnerability stemmed from the use of `assert` statements for security checks, specifically in the `get_act_fn` function. When Python is run with optimizations (e.g., `python -O`), `assert` statements are stripped out, effectively disabling the security check that restricted activation function loading to `torch.nn.modules.`. This allowed an attacker to specify an arbitrary module path.

### Impact

An attacker could achieve arbitrary code execution by supplying a specially crafted activation function name that points to a malicious module or function outside of the intended `torch.nn.modules.` namespace.

### Key diff

```

-        assert function_name.startswith("torch.nn.modules."), (
-            "Loading of activation functions is restricted to "
-            "torch.nn.modules for security reasons"
-        )
+        if not function_name.startswith("torch.nn.modules."):
+            raise ValueError(
+                "Loading of activation functions is restricted to "
+                "torch.nn.modules for security reasons"
+            )
         fn = resolve_obj_by_qualname(function_name)()
```

### Fix summary

The patch replaces all security-critical `assert` statements with explicit `if` conditions that raise `ValueError` or `RuntimeError` exceptions. This ensures that the security checks are always active, regardless of Python's optimization settings, preventing the bypass of restrictions on loading activation functions.

## GHSA-534h-c3cw-v3h9 · nuxt

2026-06-16 · JavaScript · **MODERATE** 5.5 · Pattern: `INSECURE_DEFAULT→CONFIG`

### Root cause

The Nuxt dev server's vite-node IPC socket on Linux was created in the system's temporary directory with default permissions, making it world-connectable. This allowed any local user to connect to the socket and potentially interact with the dev server.

### Impact

A local attacker could connect to the IPC socket and send arbitrary commands to the Nuxt dev server, potentially leading to information disclosure, code execution, or other unauthorized actions within the context of the dev server process.

### Key diff

### Fix summary

The patch changes how the IPC socket path is generated and handled on non-Windows systems. It now creates a temporary directory with 0700 permissions (owner-only access) and places the socket inside it. Additionally, after the socket is created and listened to, its permissions are explicitly set to 0600 (owner read/write only) to further restrict access. The `umask` is temporarily set to 0o077 during socket creation to prevent brief exposure with default permissions.

## GHSA-934w-87qh-qr26 · nuxt

2026-06-16 · JavaScript · **MODERATE** 0.0 · Pattern: `UNSANITIZED_INPUT→XSS`

## GHSA-c9cv-mq2m-ppp3 · nuxt

2026-06-16 · JavaScript · **MODERATE** 0.0 · Pattern: `OPEN_REDIRECT→PHISHING`

### Root cause

The application did not properly sanitize or validate user-supplied URLs before using them in redirects or rendering them in HTML attributes. ## GHSA-mm7m-92g8-7m47 · nuxt

2026-06-16 · JavaScript · **HIGH** 0.0 · Pattern: `MISSING_AUTHZ→RESOURCE`

### Root cause

The vulnerability stemmed from a case-sensitivity mismatch in how Nuxt's route rules matcher processed incoming paths compared to how vue-router handled them. The route rules matcher did not consistently convert paths to lowercase, allowing an attacker to bypass security middleware by using a different casing for the URL than what was configured in the route rules.

### Impact

An attacker could bypass security middleware, authentication checks, or other access control mechanisms defined in `routeRules` by manipulating the casing of the URL path. This could lead to unauthorized access to protected routes or resources.

### Key diff

```
-      export default (path) => defu({}, ...matcher('', path).map(r => r.data).reverse())
+      export default (path) => defu({}, ...matcher('', typeof path === 'string' ? path.toLowerCase() : path).map(r => r.data).reverse())
```
```diff
-    return routeRulesMatcher(path)
+    return routeRulesMatcher(path.toLowerCase())
```

### Fix summary

The patch ensures that the `routeRulesMatcher` consistently converts the incoming path to lowercase before matching against defined route rules. This standardizes the path comparison, preventing case-sensitive bypasses and ensuring that security middleware is correctly applied.
