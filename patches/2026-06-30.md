## GHSA-m63v-2g9w-2w6v · github.com/fission/fission

2026-06-30 · Go · **CRITICAL** 9.9 · Pattern: `PRIVILEGE_ESCALATION→ROLE`

### Root cause

The Fission platform allowed users to specify container configurations for environments (Runtime.Container and Builder.Container) that were not subject to the same security context validation as standard PodSpecs. This oversight meant that dangerous security settings like 'privileged=true' or 'allowPrivilegeEscalation=true' could be set in these specific container fields, bypassing existing security checks.

### Impact

An attacker could create privileged pods within the Kubernetes cluster, effectively escaping the container sandbox and gaining root-level access to the host or other cluster resources, leading to full cluster compromise.

### Key diff

```
--- a/pkg/apis/core/v1/validation.go
+++ b/pkg/apis/core/v1/validation.go
 	errs = errors.Join(errs, ValidatePodSpecSafety("Environment.spec.runtime.podspec", e.Spec.Runtime.PodSpec))
 	errs = errors.Join(errs, ValidatePodSpecSafety("Environment.spec.builder.podspec", e.Spec.Builder.PodSpec))
+	errs = errors.Join(errs, ValidateContainerSafety("Environment.spec.runtime.container", e.Spec.Runtime.Container))
+	errs = errors.Join(errs, ValidateContainerSafety("Environment.spec.builder.container", e.Spec.Builder.Container))
 	return errs
```

### Fix summary

The patch introduces a new `ValidateContainerSafety` function to explicitly check the security context of individual containers, specifically applying it to the previously unchecked `Runtime.Container` and `Builder.Container` fields in the Environment CRD. Additionally, a sanitization step is added during container merging to strip dangerous security context settings, providing a defense-in-depth measure even if admission webhooks are bypassed.

## GHSA-7m8x-qg2j-4m3v · github.com/fission/fission

2026-06-30 · Go · **HIGH** 8.1 · Pattern: `UNCLASSIFIED`

### Root cause

The Fission controller was materializing Kubernetes Secret values directly into Deployment environment variables, making them visible in the Deployment spec. Additionally, it accepted arbitrary user-supplied PodSpec fields from MessageQueueTrigger, allowing users to potentially inject malicious configurations into the created pods.

### Impact

An attacker could read sensitive Secret values by inspecting the Deployment object or inject arbitrary PodSpec fields to gain control over the created pods, leading to privilege escalation or arbitrary code execution within the cluster.

## GHSA-8j3x-m868-cpw8 · openbabel

2026-06-30 · Python · **LOW** 7.8 · Pattern: `BUFFER_OVERFLOW→HEAP`

### Root cause

The vulnerability existed because `memcpy` was used to copy data between potentially overlapping memory regions. When the source and destination buffers overlap, `memcpy`'s behavior is undefined, which can lead to data corruption or out-of-bounds writes.

### Impact

An attacker could potentially cause a denial of service (crash) due to memory corruption or, in more complex scenarios, achieve arbitrary code execution by manipulating memory in an unexpected way.

### Key diff

```
-	memcpy(&_buffer[0] + (4 - n_putback),
-	           this->gptr() - n_putback,
-	           n_putback * sizeof(char_type));
+	memmove(&_buffer[0] + (4 - n_putback),
+	            this->gptr() - n_putback,
+	            n_putback * sizeof(char_type));
```

### Fix summary

The patch replaces the `memcpy` function call with `memmove`. `memmove` is designed to handle overlapping source and destination memory regions correctly, ensuring that the data is copied safely without undefined behavior.
