## GHSA-m8xg-8xg9-mxhm · github.com/nuclio/nuclio

2026-06-04 · Go · **HIGH** 8.3 · Pattern: `MISSING_AUTHZ→RESOURCE`

### Root cause

The Nuclio platform's project write paths (create, update, delete) lacked proper authorization checks. While some authorization mechanisms were present, they were not consistently applied or fully enforced for all project modification operations, allowing any authenticated user to bypass intended access controls.

An authenticated attacker could modify, create, or delete any project within the Nuclio environment, leading to data integrity issues, denial of service, or unauthorized resource manipulation.

## GHSA-j5f8-grm9-p9fc · axios

2026-06-04 · JavaScript · **HIGH** 7.5 · Pattern: `INFO_DISCLOSURE→ERROR_MESSAGE`

### Root cause

The Axios HTTP adapter did not properly handle the `Proxy-Authorization` header during redirects. When a request was redirected from a proxy-enabled connection to a direct connection, or to a different proxy, the `Proxy-Authorization` header from the initial request was inadvertently carried over to the subsequent redirect target.

### Impact

An attacker could potentially leak sensitive `Proxy-Authorization` credentials to an unintended third-party server if a redirect occurred from a proxied connection to a server controlled by the attacker, or to a server that does not require the same proxy.

## GHSA-86qp-5c8j-p5mr · starlette

2026-06-04 · Python · **MODERATE** 6.5 · Pattern: `UNSANITIZED_INPUT→HEADER`

### Root cause

The Starlette framework's URL construction logic did not properly validate the 'Host' header. If a malicious 'Host' header contained characters like '/', '?', or '#', it could be used to poison the `request.url.path` attribute, making it differ from the actual `scope["path"]`.

### Impact

An attacker could manipulate the `request.url.path` to bypass security checks that rely on path-based validation, potentially leading to unauthorized access or other security policy violations.

### Fix summary

The patch introduces a regular expression (`_HOST_RE`) to strictly validate the 'Host' header. It ensures that the 'Host' header only contains valid characters (alphanumeric, hyphens, dots, and square brackets for IPv6, with an optional port) before being used in URL construction, preventing malicious path injection.

## GHSA-fh3h-vg37-cc95 · webob

2026-06-04 · Python · **MODERATE** 6.1 · Pattern: `OPEN_REDIRECT→PHISHING`

### Root cause

The vulnerability stemmed from the `urlparse.urljoin` function's behavior when processing a location header that started with `//`. Instead of treating `//example.com` as a relative path, `urlparse.urljoin` would interpret it as an absolute URL with a scheme-relative authority, leading to an unintended redirect to an external domain.

### Impact

An attacker could craft a malicious URL that, when processed by the vulnerable application, would redirect users to an arbitrary external website controlled by the attacker. This could be used for phishing attacks or to bypass security checks.

### Key diff

```
--- a/src/webob/response.py
+++ b/src/webob/response.py
@@ -1359,6 +1359,11 @@ def _make_location_absolute(environ, value):

             return value
 
+        if value.startswith("//"):
+            value = f"/%2f{value[2:]}"
         new_location = urlparse.urljoin(_request_uri(environ), value)
 
         return new_location
```

### Fix summary

The patch addresses the issue by explicitly checking if the `value` (location header) starts with `//`. If it does, it prepends `/%2f` to the value, effectively transforming `//example.com` into `/%2f/example.com`. This forces `urlparse.urljoin` to treat the path as relative to the current domain, preventing the open redirect.
