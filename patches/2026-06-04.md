## GHSA-8whc-2wmv-ww35 · WWBN/AVideo

2026-06-04 · PHP · **CRITICAL** 9.6 · Pattern: `UNSANITIZED_INPUT→XSS`

## GHSA-m8xg-8xg9-mxhm · github.com/nuclio/nuclio

2026-06-04 · Go · **HIGH** 8.3 · Pattern: `MISSING_AUTHZ→RESOURCE`

### Root cause

The Nuclio platform's project write paths (create, update, delete) lacked proper authorization checks. While some authorization mechanisms were present, they were not consistently applied or fully enforced for all project modification operations, allowing any authenticated user to bypass intended access controls.

### Impact

An authenticated attacker could modify, create, or delete any project within the Nuclio environment, leading to data integrity issues, denial of service, or unauthorized resource manipulation.

## GHSA-2gcr-mfcq-wcc3 · hono

2026-06-04 · JavaScript · **MODERATE** 5.3 · Pattern: `UNCLASSIFIED`

### Root cause

The application used `url.pathname` directly, which is already URL-decoded, to strip a mount prefix from the request path. When the original path contained percent-encoded characters, stripping the prefix from the decoded path resulted in an incorrect path being used for routing.

### Impact

An attacker could craft a URL with percent-encoded characters that would bypass intended routing logic, potentially accessing unauthorized endpoints or causing unexpected application behavior.

### Key diff

```
--- a/src/hono-base.ts
+++ b/src/hono-base.ts
@@ -364,7 +364,7 @@ class Hono<

         const url = new URL(request.url)
-        url.pathname = url.pathname.slice(pathPrefixLength) || '/'
+        url.pathname = this.getPath(request).slice(pathPrefixLength) || '/'
         return new Request(url, request)
       }
     })()
```

### Fix summary

The patch changes the method used to obtain the path for prefix stripping. Instead of using the already decoded `url.pathname`, it now uses `this.getPath(request)`, which provides the undecoded path, ensuring correct prefix removal for percent-encoded URLs.

2026-06-04 · Go · **MODERATE** 0.0 · Pattern: `UNSANITIZED_INPUT→SQL`

### Root cause

The vulnerability existed because the `jsonPath` variable, which could contain user-controlled input, was directly interpolated into the SQL query string using `fmt.Sprintf` without proper sanitization or parameterization. Although `sqlbuilder.Escape` was used, it was applied to the entire `JSON_VALUE` function call, not just the user-controlled `jsonPath` part, allowing an attacker to inject arbitrary SQL.

### Impact

An attacker could manipulate the `jsonPath` input to execute arbitrary SQL commands on the ClickHouse database, potentially leading to data exfiltration, modification, or denial of service.

### Key diff

```
-	sb.Select(fmt.Sprintf("JSON_VALUE('{}', '%s')", sqlbuilder.Escape(d.jsonPath)))
+	return sqlbuilder.Buildf("SELECT JSON_VALUE('{}', %v)", d.jsonPath).
+		BuildWithFlavor(sqlbuilder.ClickHouse)
```

### Fix summary

The patch replaces the manual string formatting with `sqlbuilder.Buildf`, which correctly parameterizes the `jsonPath` variable. This ensures that the user-controlled input is treated as a literal string within the SQL query, preventing injection.
