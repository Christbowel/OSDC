## GHSA-598g-h2vc-h5vg · github.com/juev/nebula-mesh

2026-06-08 · Go · **CRITICAL** 9.9 · Pattern: `PRIVILEGE_ESCALATION→ROLE`

### Root cause

The application used a cached context value for `actorIsAdmin` checks, which meant that if an operator's role was downgraded from 'admin' to a regular user, their active session would still incorrectly reflect them as an administrator. This allowed them to bypass authorization checks on various API endpoints.

### Impact

An attacker could maintain administrative privileges even after their role was revoked, enabling them to perform actions such as managing other operators, accessing audit logs, listing all CAs, and other sensitive operations that should be restricted to active administrators.

## GHSA-w8p2-r796-3vmq · authlib

2026-06-08 · Python · **MODERATE** 5.4 · Pattern: `OPEN_REDIRECT→PHISHING`

### Root cause

The vulnerability existed because the authorization server would unconditionally use the `redirect_uri` provided in the request payload when an `UnsupportedResponseTypeError` occurred. This allowed an attacker to control the `redirect_uri` even if the `response_type` was not supported, bypassing the validation that would normally occur for supported response types.

### Impact

An attacker could craft a malicious authorization request with an unsupported `response_type` and an arbitrary `redirect_uri`. When the server returned an error, it would redirect the user's browser to the attacker-controlled URL, enabling phishing attacks.

### Key diff

```

+++ b/authlib/oauth2/rfc6749/authorization_server.py
```

### Fix summary

The patch modifies the error handling for `UnsupportedResponseTypeError` to only use a `redirect_uri` if the `client_id` is present, the client exists, and the `redirect_uri` has been explicitly validated against the client's registered URIs. If no valid `redirect_uri` is found, the error is raised without a redirect.

## GHSA-qm33-p5p9-f8vg · github.com/juev/nebula-mesh

2026-06-08 · Go · **HIGH** 0.0 · Pattern: `MISSING_AUTHZ→RESOURCE`

### Root cause

The application lacked proper authorization checks for several host-related API endpoints. Specifically, non-admin users could list all hosts, access details of any host, and perform actions like deleting, blocking, unblocking, rotating certificates, and minting enrollment tokens for hosts they did not own or have permission to manage. The `handleListHosts` endpoint returned all hosts without filtering based on the user's permissions, and other host-specific endpoints did not verify if the requesting user had access to the target host's associated network or CA.

### Impact

An attacker with operator privileges could gain unauthorized access to information about all hosts in the system, including those belonging to other tenants or networks. They could also perform administrative actions on any host, such as deleting, blocking, or rotating certificates, leading to denial of service, unauthorized network access, or disruption of services for other users.
