## GHSA-j9jx-hp4c-ghhh · github.com/filebrowser/filebrowser

2026-06-12 · Go · **HIGH** 7.5 · Pattern: `MISSING_AUTHZ→RESOURCE`

### Root cause

When a public share was created for a subdirectory, the filesystem root for the user was rebased to that subdirectory. However, the access control rules, which are defined relative to the user's original scope, were not re-evaluated against the original path. This allowed an attacker to bypass deny rules that were intended to apply to paths within the shared directory but outside the rebased root.

### Impact

An attacker could access files or directories within a publicly shared folder that should have been restricted by access control rules, leading to unauthorized information disclosure.

## GHSA-248m-82v9-q6g6 · pypdf

2026-06-12 · Python · **MODERATE** 3.3 · Pattern: `DOS→RESOURCE_EXHAUSTION`

### Root cause

The vulnerability existed because the code did not properly handle a specific edge case in PDF cross-reference streams where the 'entry_sizes' values summed to zero. Previously, a 'max(..., 1)' was used to prevent ZeroDivisionError, but this allowed 'min_entry_bytes' to be 1 even when no entry data was encoded, leading to an extremely large 'max_entries' calculation.

### Impact

An attacker could craft a malicious PDF file with zero-only width values in the cross-reference stream. When this file is processed, it would lead to an excessively long runtime due to the calculation of a very large 'max_entries', effectively causing a denial of service for the application processing the PDF.

### Key diff

```
+            if self.strict:

+            return []
```

### Fix summary

The patch explicitly checks if 'min_entry_bytes' is zero after summing the 'entry_sizes'. If it is zero, it logs a warning (or raises a PdfStreamError in strict mode) and returns an empty list, preventing the calculation of an extremely large 'max_entries' and thus mitigating the denial of service.

## GHSA-pjpj-v387-x4vq · typo3/cms-form

2026-06-12 · PHP · **HIGH** 0.0 · Pattern: `MISSING_AUTHZ→RESOURCE`

### Root cause

The vulnerability stemmed from an insufficient check in the `generateErrorsIfFormDefinitionIsValidButHasInvalidFileExtension` method. It only threw an error if a form definition was valid but had an invalid file extension. This allowed invalid form definitions (e.g., non-YAML files) with a valid file extension to bypass the check and potentially be processed, leading to unintended access or information disclosure.

### Impact

An attacker could potentially bypass file extension checks by providing an invalid form definition (e.g., a non-YAML file) that still had the correct '.form.yaml' extension.
