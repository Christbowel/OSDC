## GHSA-jg22-mg44-37j8 · aiohttp

2026-06-03 · Python · **MODERATE** 6.4 · Pattern: `DESERIALIZATION→RCE`

### Root cause

The `aiohttp.cookiejar.CookieJar` class used Python's `pickle` module to serialize and deserialize cookie data to and from files. The `pickle.load()` function, when used with untrusted input, can lead to arbitrary code execution because it can reconstruct arbitrary Python objects, including those that can trigger malicious code during their instantiation.

### Impact

An attacker who can control the content of the cookie file (e.g., by providing a malicious file or injecting data into an existing one) could achieve arbitrary code execution on the system running the `aiohttp` application when the `CookieJar.load()` method is called.

## GHSA-ch57-39q2-4crm · malla

2026-06-03 · Python · **MODERATE** 6.3 · Pattern: `UNSANITIZED_INPUT→XSS`

### Root cause

The vulnerability existed because the `ModernTable` component in the frontend directly rendered user-controlled data (like Meshtastic node names) into the HTML without proper sanitization. Specifically, the `renderTableHeader()` method constructed HTML strings using template literals, embedding column titles directly, which allowed malicious scripts to be injected and executed when the table was rendered.

### Impact

An attacker could inject arbitrary client-side scripts into the web application. This could lead to session hijacking, defacement of the website, redirection to malicious sites, or other client-side attacks against users viewing the affected pages.

## GHSA-c27g-q93r-2cwf · vite

### Root cause

The application was vulnerable to command injection on Windows due to insufficient sanitization of user-provided file names when launching an editor. ## GHSA-hg6j-4rv6-33pg · aiohttp

2026-06-03 · Python · **MODERATE** 0.0 · Pattern: `OPEN_REDIRECT→PHISHING`

### Root cause

The aiohttp client library did not clear cookies when following a redirect to a different origin. This meant that cookies intended for the original domain could be sent to an untrusted third-party domain if a redirect occurred.

### Impact

An attacker could craft a malicious redirect to an arbitrary domain, causing the user's sensitive cookies (e.g., session tokens) to be sent to the attacker-controlled domain, leading to session hijacking or information disclosure.

### Key diff

```
if url.origin() != redirect_origin:
                             auth = None
+                            cookies = None
                             headers.pop(hdrs.AUTHORIZATION, None)
                             headers.pop(hdrs.COOKIE, None)
```

### Fix summary

The patch explicitly sets the 'cookies' variable to None when a redirect leads to a different origin. This ensures that per-request cookies are not forwarded to a new, potentially untrusted, origin during cross-origin redirects, preventing their leakage.
