## GHSA-cwc9-cp4j-mcvv · @libp2p/gossipsub

2026-07-10 · JavaScript · **HIGH** 7.5 · Pattern: `DOS→RESOURCE_EXHAUSTION`

### Root cause

The libp2p gossipsub implementation did not properly limit the size of arrays within IHAVE and IWANT control messages. An attacker could send messages with extremely large arrays of message IDs, causing the receiving node to spend excessive CPU cycles processing them without sufficient bounds checking.

### Impact

An attacker could send specially crafted oversized control messages to a victim node, leading to high CPU utilization and potentially a denial of service for that node, disrupting its ability to participate in the network.

### Key diff

```
--- a/packages/gossipsub/src/message/decodeRpc.ts
+++ b/packages/gossipsub/src/message/decodeRpc.ts
@@ -9,11 +9,12 @@ export interface DecodeRPCLimits {
 }
 
 export const defaultDecodeRpcLimits: DecodeRPCLimits = {
-  maxSubscriptions: Infinity,
-  maxMessages: Infinity,
-  maxIhaveMessageIDs: Infinity,
-  maxIwantMessageIDs: Infinity,
-  maxIdontwantMessageIDs: Infinity,
-  maxControlMessages: Infinity,
-  maxPeerInfos: Infinity
+  maxSubscriptions: 5000,
+  maxMessages: 5000,
+  maxIhaveMessageIDs: 5000,
+  maxIwantMessageIDs: 5000,
+  maxControlMessages: 5000,
+  maxIdontwantMessageIDs: 512,
+  maxPeerInfos: 16
 }
```

### Fix summary

The patch introduces explicit limits for the maximum number of subscriptions, messages, and message IDs within IHAVE and IWANT control messages. It also adds flood protection for IWANT messages by tracking the number of requests from a peer within a heartbeat interval and ignoring excessive requests.
