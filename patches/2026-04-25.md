## GHSA-57r2-h2wj-g887 · openclaw

2026-04-25 · JavaScript · **LOW** 0.0 · Pattern: `UNCLASSIFIED`

### Root cause

The system was incorrectly recording awareness events from isolated cron jobs as 'trusted' system events. This meant that events originating from potentially untrusted or less privileged contexts were being logged with the same trust level as internal, trusted system events, blurring the lines of event provenance and integrity.

### Impact

An attacker could potentially inject misleading or untrusted information into system event logs, making it harder for administrators to distinguish legitimate system events from those originating from isolated, potentially compromised, or untrusted cron jobs. This could aid in obfuscating malicious activity or confusing incident response.

### Fix summary

The patch introduces a 'trusted' flag to the `enqueueSystemEvent` function. For awareness events originating from isolated cron agents, this flag is explicitly set to `false`, ensuring that these events are correctly marked as untrusted when recorded. This clarifies the origin and trust level of logged events.

## GHSA-72q8-jcmc-97wx · openclaw

2026-04-25 · JavaScript · **MODERATE** 0.0 · Pattern: `MISSING_AUTHZ→RESOURCE`

### Root cause

The system incorrectly determined the chat type (P2P vs. group) for Feishu card actions. It relied on a fallback logic that assumed a chat_id implied a group chat, and otherwise P2P. This misclassification allowed certain actions intended for group chats to be processed as P2P (Direct Message) interactions, bypassing specific DM policies.

### Impact

An attacker could potentially bypass direct message (DM) policies by crafting a card action that is misclassified as a P2P message, even if it originated from a group chat, leading to unauthorized actions or information disclosure.

### Key diff

```
+            log,
+          }),
```

### Fix summary

The patch introduces a new function, `resolveCardActionChatType`, which explicitly queries the Feishu API (`im.chat.get`) to accurately determine the chat type (p2p or group) based on `chat_mode` or `chat_type` fields. It also adds caching for resolved chat types to improve performance and ensures that the `chatType` parameter is always explicitly passed and resolved, removing the ambiguous fallback logic.
