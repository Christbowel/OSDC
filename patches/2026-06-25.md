## GHSA-2933-q333-qg83 · i18next-fs-backend

2026-06-25 · JavaScript · **CRITICAL** 9.1 · Pattern: `PROTOTYPE_POLLUTION→OVERRIDE`

### Root cause

The application's utility functions `getLastOfPath`, `setPath`, and `pushPath` did not properly sanitize user-controlled input used as object keys. This allowed an attacker to inject special keys like `__proto__`, `constructor`, or `prototype` into the object path, leading to modification of `Object.prototype`.

### Impact

An attacker could modify the properties of `Object.prototype`, which could lead to denial of service, remote code execution, or other severe impacts depending on the application's usage of JavaScript objects and their properties.

### Fix summary

The patch introduces a check for 'UNSAFE_KEYS' (e.g., `__proto__`, `constructor`, `prototype`) within the `getLastOfPath` function. If an unsafe key is detected, the function now returns an empty object or `undefined`, preventing the traversal into or modification of `Object.prototype`.

## GHSA-f49m-vf83-692w · i18next-http-middleware

2026-06-25 · JavaScript · **CRITICAL** 9.1 · Pattern: `PROTOTYPE_POLLUTION→OVERRIDE`

### Root cause

The `missingKeyHandler` in `i18next-http-middleware` did not adequately sanitize incoming keys from user requests. While it had a denylist for literal unsafe keys like `__proto__`, it failed to account for keys containing dotted segments (e.g., `__proto__.polluted`). When these keys were processed by backends that split them using a `keySeparator` (like `i18next-fs-backend`), the individual segments could then be used in an unguarded `setPath` operation, leading to prototype pollution.

### Impact

An attacker could manipulate object prototypes, potentially leading to denial of service, remote code execution in certain contexts, or other unexpected application behavior by injecting properties into `Object.prototype`.
