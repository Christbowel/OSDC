## GHSA-jjrm-hr5f-673x · github.com/fluxcd/source-controller

2026-06-05 · Go · **MODERATE** 0.0 · Pattern: `PATH_TRAVERSAL→FILE_READ`

### Root cause

The application was using `filepath.Join` to construct file paths based on user-controlled input (sparse checkout paths and object keys). `filepath.Join` does not sanitize or validate path components, allowing an attacker to use `../` sequences to traverse outside the intended base directory.

### Impact

An attacker could read, write, or delete arbitrary files on the system where the controller is running, depending on the context and permissions of the process. This could lead to information disclosure, data corruption, or even remote code execution.

### Fix summary

The patch replaces `filepath.Join` with `securejoin.SecureJoin` when constructing paths from user-controlled input. `securejoin.SecureJoin` explicitly checks for and prevents path traversal sequences like `../`, ensuring that the resulting path remains within the designated base directory.

## GHSA-wx3m-whqv-xv47 · skillctl

2026-06-05 · RUST · **HIGH** 0.0 · Pattern: `PATH_TRAVERSAL→FILE_READ`

### Root cause

The application was vulnerable to path traversal because it did not properly validate user-controlled paths (source_path and destination) read from the .skills.toml configuration file. This allowed paths containing '..' or absolute paths to be processed, potentially leading to operations outside the intended project directory.

### Impact

An attacker could craft a malicious .skills.toml file to read or delete arbitrary files on the system where the skillctl tool is run, leading to information disclosure or denial of service.

### Key diff

```

+++ b/src/project_config.rs
@@ -21,6 +24,31 @@ pub struct InstalledSkill {

+impl InstalledSkill {
+    pub fn validate(&self) -> Result<(), AppError> {
+        validate_relative_subpath(&self.source_path).map_err(|e| {
+            AppError::Config(format!(
+                "invalid source_path for skill `{}` in .skills.toml: {e}",
+                self.name
+            ))
+        })?;
+        validate_relative_subpath(&self.destination).map_err(|e| {
+            AppError::Config(format!(
+                "invalid destination for skill `{}` in .skills.toml: {e}",
+                self.name
+            ))
+        })?;
+        Ok(())
+    }
+}
```

### Fix summary

A new module `path_safety.rs` was introduced to provide lexical path validation. This validation rejects any path components that are absolute (`RootDir`, `Prefix`) or traverse parent directories (`ParentDir`). The `InstalledSkill` struct now includes a `validate` method that uses this new path safety logic to check `source_path` and `destination` fields when loading the project configuration.
