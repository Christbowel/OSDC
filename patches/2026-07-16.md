## GHSA-hvrp-rf83-w775 · mcp

2026-07-16 · Python · **HIGH** 7.6 · Pattern: `IDOR→DATA_ACCESS`

### Root cause

The experimental task handlers in the MCP Python SDK did not properly associate tasks with the sessions that created them. When a task was created, its ID was not scoped to the client's session, meaning any client possessing a task ID could access or cancel tasks created by other clients.

### Impact

An attacker could enumerate, access, or cancel tasks belonging to other clients, leading to unauthorized data access, denial of service, or disruption of legitimate client operations.

### Key diff

```
--- a/src/mcp/server/experimental/task_support.py
+++ b/src/mcp/server/experimental/task_support.py
@@ -91,12 +92,22 @@ def configure_session(self, session: ServerSession) -> None:

+        It also assigns the session a task session scope. Task IDs generated
+        by `run_task()` embed this scope, and the default task handlers only
+        operate on tasks created by the requesting session. Stateless sessions
+        are not assigned a scope: each request runs on a fresh session, so a
+        task created by one request could never be retrieved by a later one if
+        tasks were bound to the session that created them.
```

### Fix summary

The patch introduces session-scoped task IDs. Task IDs generated by `run_task()` now embed an opaque, per-session marker. Default task handlers are updated to only operate on tasks created by the requesting session, preventing unauthorized access or cancellation by other clients.

## GHSA-jpw9-pfvf-9f58 · mcp

2026-07-16 · Python · **HIGH** 7.1 · Pattern: `MISSING_AUTH→ENDPOINT`

### Root cause

The MCP Python SDK's HTTP transports were serving session requests without properly verifying the authenticated principal. This means that while a client ID might have been present, the actual user (subject) associated with the session was not consistently propagated or verified across the authentication flow (authorization code, refresh token, access token), leading to a potential disconnect between the authenticated client and the actual resource owner.

### Impact

An attacker could potentially manipulate or bypass authentication checks by exploiting the lack of principal verification, possibly gaining unauthorized access to resources or impersonating other users if the system relied solely on client ID without linking it to a specific subject.

### Key diff

```
--- a/src/mcp/server/auth/provider.py
+++ b/src/mcp/server/auth/provider.py

+    subject: str | None = None  # resource owner; propagate to the issued AccessToken

```

### Fix summary

The patch introduces a 'subject' field to AuthorizationCode, RefreshToken, and AccessToken models. This 'subject' field, representing the resource owner, is now propagated throughout the authentication flow, from the initial authorization code request to the issued and refreshed access tokens. Additionally, the 'subject' and 'issuer' claims are explicitly added to the introspection response and token verification process.

## GHSA-46wq-28cx-mhw4 · nimiq-primitives

2026-07-16 · RUST · **LOW** 3.7 · Pattern: `DOS→RESOURCE_EXHAUSTION`

### Root cause

The `child_index` function in `TrieProofNode` used `unwrap()` on the result of `child_prefix.get(self.key.len())`. This `unwrap()` would panic if `child_prefix` had the same length as `self.key`, which was not strictly a prefix. This allowed an attacker to craft a malicious `TrieProof` with equal-length keys, leading to a panic and thus a denial of service.

### Impact

An attacker could provide a specially crafted Merkle Radix Trie proof that would cause the node to panic and crash, leading to a denial of service for the application or service using the `nimiq-primitives` library.

## GHSA-vj7q-gjh5-988w · mcp

2026-07-16 · Python · **HIGH** 0.0 · Pattern: `CORS_MISCONFIGURATION→ORIGIN`

### Root cause

The WebSocket server transport in the MCP Python SDK did not perform any validation on the Host or Origin headers of incoming WebSocket handshake requests. This allowed an attacker to connect to the WebSocket server from an arbitrary origin, bypassing typical browser-based same-origin policy protections.

### Impact

An attacker could potentially craft malicious web pages to interact with the WebSocket server, leading to Cross-Site WebSocket Hijacking (CSWSH) or DNS rebinding attacks if the server was intended to be accessed only from specific origins.
