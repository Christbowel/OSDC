## GHSA-jxpj-9j24-w337 · com.ctrip.framework.apollo:apollo

2026-07-13 · Java · **MODERATE** 6.5 · Pattern: `MISSING_AUTHZ→RESOURCE`

### Root cause

The application was missing an authorization check for accessing release configurations. While an authentication mechanism might have been in place, it did not verify if the authenticated user had the necessary permissions to view specific configuration releases, allowing any authenticated user to potentially access sensitive configuration data.

### Impact

An attacker could gain unauthorized access to sensitive configuration information, potentially leading to further compromise of the system or disclosure of confidential data.

### Key diff

```
if (release == null) {
      throw NotFoundException.releaseNotFound(releaseId);
    }
+    if (userPermissionValidator.shouldHideConfigToCurrentUser(release.getAppId(), env,
+        release.getClusterName(), release.getNamespaceName())) {
+      throw new AccessDeniedException("Access is denied");
+    }
```

### Fix summary

The patch introduces a new authorization check using `userPermissionValidator.shouldHideConfigToCurrentUser` before returning the release configuration. If the current user does not have the required permissions for the specific application, environment, cluster, and namespace, an `AccessDeniedException` is thrown.

## GHSA-rwcv-whm8-fmxm · geonode

2026-07-13 · Python · **MODERATE** 6.1 · Pattern: `UNSANITIZED_INPUT→XSS`

### Root cause

The application was rendering user-supplied HTML content directly into templates using the `|safe` filter without prior sanitization. This allowed attackers to inject malicious scripts into fields like 'abstract', 'keywords', 'restrictions', 'purpose', 'data_quality_statement', and 'supplemental_information' when creating or editing resources.

### Impact

An attacker could inject arbitrary JavaScript, leading to session hijacking, defacement, or full account takeover if an administrator or other privileged user viewed the compromised content.

### Key diff

```
--- a/geonode/templates/metadata_detail.html
+++ b/geonode/templates/metadata_detail.html
@@ -45,7 +46,7 @@
         {% block abstract %}
         {% if resource.abstract %}
         <dt>{% trans "Abstract" %}</dt>
-        <dd>{{ resource.abstract|safe }}</dd>
+        <dd>{{ resource.abstract|sanitize_html|safe }}</dd>
         {% endif %}
         {% endblock abstract %}
```

### Fix summary

A new Django template filter `sanitize_html` was introduced, which uses the `nh3` library to safely clean HTML content. This filter is now applied to all user-controlled fields in `metadata_detail.html` before being marked as safe, ensuring that dangerous tags and attributes are removed.
