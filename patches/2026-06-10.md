## GHSA-5g86-85rp-f9hx · @papra/webhooks

2026-06-10 · JavaScript · **LOW** 3.5 · Pattern: `SSRF→INTERNAL_ACCESS`

### Root cause

The `ofetch` library, used for making HTTP requests in the webhook delivery system, by default follows redirects. This behavior was not explicitly disabled or controlled. An attacker could provide a malicious URL that redirects to an internal resource, bypassing the existing SSRF protection which only checked the initial URL.

### Impact

An attacker could bypass the SSRF protection and make the application send requests to internal network resources or cloud metadata endpoints, potentially leading to information disclosure or further attacks.

### Key diff

```

+++ b/packages/webhooks/src/webhooks.services.ts

   const response = await ofetch.raw<unknown>(url, {
     ...options,
     ignoreResponseError: true,
+    redirect: 'manual',
   });
 
   return {
```

### Fix summary

The patch introduces a custom HTTP client for webhooks that explicitly disables redirects for the `ofetch` library by setting `redirect: 'manual'`. This ensures that the application does not automatically follow redirects to potentially malicious internal URLs, thus enforcing the SSRF protection on all subsequent redirect locations.

## GHSA-8h84-fhqq-q58v · github.com/juev/nebula-mesh

2026-06-10 · Go · **MODERATE** 0.0 · Pattern: `INFO_DISCLOSURE→STACK_TRACE`

### Root cause

The application decrypted a CA private key into memory (keyBytes) but did not explicitly zeroize or clear this sensitive data from the heap if an error occurred during the subsequent CA manager construction. This left the plaintext private key lingering in memory, potentially accessible.

### Impact

An attacker with memory access (e.g., via a memory dump or other memory-reading vulnerability) could potentially retrieve the decrypted CA private key from the heap, leading to compromise of the CA.

### Key diff

```
--- a/internal/pki/resolver.go
+++ b/internal/pki/resolver.go
-	return LoadCAFromMaterial([]byte(c.CertPEM), ed25519.PrivateKey(keyBytes))
+	return loadCAOrZeroize([]byte(c.CertPEM), keyBytes)
+
+// loadCAOrZeroize builds a CAManager from freshly-decrypted key material,
+// transferring ownership of keyBytes to the manager on success (the caller
+// later clears it via CAManager.Wipe). If construction fails — e.g. the cert
+// is unparseable or not a CA — the plaintext key never reaches a manager, so
+// it is zeroized here rather than left on the heap for the GC (#181).
+func loadCAOrZeroize(certPEM, keyBytes []byte) (*CAManager, error) {
+	mgr, err := LoadCAFromMaterial(certPEM, ed25519.PrivateKey(keyBytes))
+	if err != nil {
+		keystore.Zeroize(keyBytes)
+		return nil, err
+	}
+	return mgr, nil
```

### Fix summary

The patch introduces a new helper function `loadCAOrZeroize` which ensures that if the `LoadCAFromMaterial` call fails, the decrypted private key bytes (`keyBytes`) are explicitly zeroized using `keystore.Zeroize` before returning an error. This prevents the sensitive key material from persisting in memory unnecessarily.
