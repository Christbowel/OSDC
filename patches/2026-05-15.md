## GHSA-3g8v-8r37-cgjm · github.com/dunglas/frankenphp

2026-05-15 · Go · **HIGH** 8.1 · Pattern: `UNCLASSIFIED`

### Root cause

The application incorrectly handled Unicode characters when splitting CGI paths. It used `golang.org/x/text/search` for case-insensitive matching, which could interpret non-ASCII Unicode characters as equivalent to ASCII characters. This allowed an attacker to craft a filename with Unicode characters that would bypass the intended `.php` extension check.

### Impact

An attacker could upload a file with a specially crafted name containing Unicode characters (e.g., `file.pHp` where 'H' is a fullwidth character) and have it executed as a PHP script, leading to arbitrary code execution on the server.

## GHSA-27w2-87xv-37c6 · nimiq-keys

2026-05-15 · RUST · **HIGH** 7.5 · Pattern: `NULL_DEREF→CRASH`

### Root cause

The `Ed25519PublicKey::verify` method directly called `Ed25519Signature::from_bytes(sig).unwrap()` without checking if the conversion was successful. If the input signature `sig` had an invalid length, `from_bytes` would return an `Err`, which `unwrap()` would then panic on, leading to a node crash.

### Impact

An attacker could send a malformed Ed25519 signature to a remote node, causing it to panic and crash. This could be used to disrupt the network via a denial-of-service attack.

+            return false;

### Fix summary

The patch modifies the `Ed25519PublicKey::verify` method to gracefully handle invalid signature lengths. Instead of calling `unwrap()`, it now uses `let Ok(signature) = ... else { return false; };` to check the result of `from_bytes`. If the conversion fails, it returns `false`, indicating an invalid signature, rather than panicking.

## GHSA-vfvv-c25p-m7mm · rkyv

2026-05-15 · RUST · **MODERATE** 0.0 · Pattern: `USE_AFTER_FREE→MEMORY`

### Root cause

The `clear` methods in `InlineVec` and `SerVec` would iterate through elements and drop them while `self.len` still reflected the original length. If a destructor panicked during this process, `self.len` would not be reset to 0, leading to a double-free or use-after-free when the vector was dropped or cleared again, potentially enabling arbitrary code execution.

### Impact

An attacker could trigger a panic in a destructor during a clear operation, leading to memory corruption (double-free or use-after-free) and potentially achieving arbitrary code execution.
