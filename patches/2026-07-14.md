## GHSA-cm26-5974-52h8 · github.com/forgekeep/nebula-mesh

2026-07-14 · Go · **HIGH** 8.1 · Pattern: `MISSING_VERIFICATION→SIGNATURE`

### Root cause

The Nebula mesh network configuration generation process did not include a mechanism to distribute and enforce certificate revocation lists (CRLs) or blocklists to individual peers. Consequently, even if a certificate was marked as revoked in the central store, the mesh nodes would continue to accept connections from peers presenting that revoked certificate.

### Impact

An attacker could continue to use a compromised or revoked certificate to access the mesh network, impersonate legitimate peers, and potentially bypass security controls, leading to unauthorized access and data breaches.

### Key diff

```
--- a/internal/configgen/marshal.go
+++ b/internal/configgen/marshal.go
 	CA   any `yaml:"ca"`
 	Cert any `yaml:"cert"`
 	Key  any `yaml:"key"`
+	Blocklist []safeString `yaml:"blocklist,omitempty"`
 }
```

### Fix summary

The patch introduces a 'Blocklist' field to the PKI configuration, which is populated with revoked certificate fingerprints from the data store. This blocklist is then included in the generated configuration for each peer. Additionally, the system now bumps the network's configuration version whenever a certificate is added to the blocklist, ensuring all peers receive the updated configuration with the revocation information.
