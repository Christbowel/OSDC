## GHSA-jvc5-6g7q-c843 · pheditor/pheditor

2026-06-09 · PHP · **CRITICAL** 9.9 · Pattern: `UNSANITIZED_INPUT→COMMAND`

### Root cause

The application was directly embedding user-supplied input from the 'dir' parameter into a shell command without proper sanitization. This allowed an attacker to inject arbitrary shell commands by manipulating the 'dir' value.

### Impact

An attacker could execute arbitrary operating system commands on the server, leading to full system compromise, data exfiltration, or denial of service.

### Key diff

```
-                $output = shell_exec((empty($dir) ? null : 'cd ' . $dir . ' && ') . $command . ' && echo \ ; pwd');
+                $output = shell_exec((empty($dir) ? null : 'cd ' . escapeshellarg($dir) . ' && ') . $command . ' && echo \ ; pwd');
```

### Fix summary

The patch addresses the vulnerability by wrapping the user-supplied 'dir' parameter with `escapeshellarg()` before it is used in the `shell_exec()` function. This ensures that any special characters in the 'dir' value are properly escaped, preventing command injection.

## GHSA-7qjx-gp9h-65qj · github.com/dexidp/dex

2026-06-09 · Go · **HIGH** 8.7 · Pattern: `MISSING_AUTHZ→RESOURCE`

## GHSA-833p-95jq-929q · phoenix_storybook

2026-06-09 · Erlang · **HIGH** 0.0 · Pattern: `DOS→RESOURCE_EXHAUSTION`

### Root cause

The application allowed user-controlled input from LiveView event parameters to be converted directly into Elixir atoms using `String.to_atom/1` or `String.to_existing_atom/1` without proper validation against a predefined set of allowed atoms. This meant an attacker could supply arbitrary strings, leading to the creation of new atoms in the Erlang VM's atom table.

### Impact

An attacker could exhaust the Erlang VM's atom table, causing the application to crash and become unavailable, leading to a denial of service for all users.

### Key diff

```

+++ b/lib/phoenix_storybook/helpers/extra_assigns_helpers.ex
@@ -172,8 +172,8 @@ defmodule PhoenixStorybook.ExtraAssignsHelpers do
   defp to_value("nil", _attr_id, _attributes, _context), do: nil
 
   defp to_value(val, attr_id, attributes, context) when is_binary(val) do
-    case declared_attr_type(attr_id, attributes) do
-      :atom -> val |> existing_value_atom!(context) |> check_type!(:atom, context)
+    case declared_attr(attr_id, attributes) do
+      %Attr{type: :atom, values: values} -> val |> to_atom_value(values, context) |> check_type!(:atom, context)

       :integer -> val |> Integer.parse() |> check_type!(:integer, context)
       :float -> val |> Float.parse() |> check_type!(:float, context)
```

### Fix summary

The patch introduces explicit validation for atom types. Instead of directly converting arbitrary strings to atoms, it now checks if the input string corresponds to an atom from a predefined list of allowed values or if it's an existing atom. New helper functions like `to_atom_value/3` and `tab_from_param/2` were added to enforce this validation and prevent unbounded atom creation.

## GHSA-mrhx-6pw9-q5fh · phoenix_storybook

2026-06-09 · Erlang · **LOW** 0.0 · Pattern: `MISSING_VERIFICATION→SIGNATURE`

### Root cause

The application allowed a PubSub topic to be directly specified via a URL parameter without any validation or verification. This meant an attacker could inject an arbitrary topic name, potentially interfering with other sessions or gaining unauthorized access to messages.

### Impact
