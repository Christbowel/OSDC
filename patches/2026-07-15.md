## GHSA-97jw-64cj-jc58 · view_component

2026-07-15 · Ruby · **HIGH** 8.7 · Pattern: `UNSANITIZED_INPUT→XSS`

### Root cause

The `around_render` callback in ViewComponent allowed developers to modify the rendered output. If this callback returned a new string object that was not explicitly marked as HTML-safe, it would bypass the automatic HTML escaping applied to the component's original template output. This meant an attacker could inject unescaped HTML, including script tags, if they could control the output of an `around_render` method.

### Impact

An attacker could achieve Cross-Site Scripting (XSS) by injecting arbitrary HTML and JavaScript into the rendered page, leading to session hijacking, defacement, or redirection.

### Key diff

Additionally, `ViewComponent::Collection` now uses `safe_join` instead of `html_safe` to concatenate rendered components, ensuring proper escaping.

## GHSA-wjhr-76vg-2hvc · garminconnect

2026-07-15 · Python · **HIGH** 8.4 · Pattern: `INSECURE_DEFAULT→CONFIG`

### Root cause

The application previously wrote sensitive Garmin OAuth tokens to disk using methods that were susceptible to the process's umask. This could result in the token file and its parent directory having overly permissive permissions, making the tokens accessible to other users on a shared system.

### Impact

An attacker with local access to the system could read the Garmin OAuth refresh token, gaining persistent access to the victim's Garmin account without needing their credentials.

### Key diff

```
--- a/garminconnect/client.py
+++ b/garminconnect/client.py
-        p.parent.mkdir(parents=True, exist_ok=True)
-        p.write_text(self.dumps())
+        p.parent.mkdir(mode=0o700, parents=True, exist_ok=True)
+        with contextlib.suppress(OSError):
+            p.parent.chmod(0o700)
+        flags = os.O_WRONLY | os.O_CREAT | os.O_TRUNC
+        if hasattr(os, "O_NOFOLLOW"):
+            flags |= os.O_NOFOLLOW
+        fd = os.open(p, flags, 0o600)
+        with os.fdopen(fd, "w", encoding="utf-8") as token_file:
+            token_file.write(self.dumps())

+            p.chmod(0o600)
```

### Fix summary

The patch explicitly sets the permissions of the token file and its parent directory to 0o600 (owner read/write) and 0o700 (owner read/write/execute) respectively. It uses `os.mkdir` with a specific mode, `os.chmod` to enforce permissions, and `os.open` with `O_CREAT` and `O_NOFOLLOW` to ensure the file is created with the correct permissions regardless of the umask.

## GHSA-q78p-hj9h-5466 · fiftyone

2026-07-15 · Python · **MODERATE** 6.3 · Pattern: `CORS_MISCONFIGURATION→ORIGIN`

### Root cause

The FiftyOne App server was configured with a wildcard Access-Control-Allow-Origin header in its CORS middleware and directly in media file response headers. This allowed any origin to make cross-origin requests to the server, bypassing same-origin policy restrictions.

### Impact

An attacker could host a malicious website and use JavaScript to make requests to the FiftyOne server, reading sensitive data from the local server that would normally be protected by the browser's same-origin policy.

### Key diff

```
--- a/fiftyone/server/app.py
+++ b/fiftyone/server/app.py
-            allow_origins=["*"]
+            allow_origins=_allowed_origins,

--- a/fiftyone/core/config.py
+++ b/fiftyone/core/config.py
+        self.allowed_origins = self.parse_string(
+            d,
+            "allowed_origins",
```

### Fix summary

The patch removes the hardcoded wildcard CORS headers from media responses and introduces a configurable `allowed_origins` setting. The CORS middleware now uses this setting, defaulting to no allowed origins if not specified, thus restricting cross-origin requests to explicitly permitted domains.

## GHSA-r3hx-x5rh-p9vv · django-haystack

2026-07-15 · Python · **HIGH** 0.0 · Pattern: `DESERIALIZATION→RCE`

### Root cause

The vulnerability stemmed from the use of `eval()` to deserialize values from Elasticsearch results. The `eval()` function executes arbitrary Python code from a string, making it unsafe when processing untrusted input.

### Impact

An attacker could inject malicious Python code into Elasticsearch results, which would then be executed on the server when the results were deserialized, leading to remote code execution.

### Key diff

```
-            converted_value = eval(value)
+            converted_value = ast.literal_eval(value)
```

### Fix summary

The patch replaces the dangerous `eval()` function with `ast.literal_eval()`. This change ensures that only safe Python literal structures (strings, numbers, tuples, lists, dicts, booleans, and None) can be evaluated, preventing the execution of arbitrary code.
