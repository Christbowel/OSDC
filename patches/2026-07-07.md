## GHSA-6qcr-qxgr-m7fv · github.com/QuantumNous/new-api

2026-07-07 · Go · **HIGH** 7.7 · Pattern: `SSRF→INTERNAL_ACCESS`

### Root cause

The application's existing Server-Side Request Forgery (SSRF) protection mechanism did not properly apply IP filtering for domain names by default. This allowed an attacker to bypass the protection by providing a hostname that would resolve to an internal IP address, as the IP-based filtering was not consistently applied to domain resolutions.

### Impact

An attacker could bypass the SSRF protection to make requests to internal network resources, potentially leading to information disclosure, access to sensitive services, or further attacks within the internal network.

### Key diff

```
Before:
	resp, err := httpClient.Get(midjourneyTask.ImageUrl)

After:
	fetchSetting := system_setting.GetFetchSetting()
	if err := common.ValidateURLWithFetchSetting(midjourneyTask.ImageUrl, fetchSetting.EnableSSRFProtection, fetchSetting.AllowPrivateIp, fetchSetting.DomainFilterMode, fetchSetting.IpFilterMode, fetchSetting.DomainList, fetchSetting.IpList, fetchSetting.AllowedPorts, fetchSetting.ApplyIPFilterForDomain); err != nil {
		c.JSON(http.StatusForbidden, gin.H{
			"error": fmt.Sprintf("request blocked: %v", err),
		})
		return
	}
	resp, err := httpClient.Get(midjourneyTask.ImageUrl)
```

### Fix summary

The patch introduces a new `ValidateURLWithFetchSetting` function that is called before making external requests. It also changes the default value of `ApplyIPFilterForDomain` to `true`, ensuring that IP filtering is applied when resolving domain names, thus preventing SSRF bypasses via unresolved hostnames. Additionally, an authentication middleware was added to a sensitive endpoint.

## GHSA-q855-8rh5-jfgq · ha-mcp

2026-07-07 · Python · **MODERATE** 6.5 · Pattern: `MISSING_AUTH→ENDPOINT`

### Root cause

The application exposed sensitive settings and policy routes at its bare root path without requiring any authentication. While Home Assistant's ingress was designed to proxy requests from a specific internal IP, direct access to the add-on's published port (9583) from other sources (e.g., LAN, reverse proxies) could bypass this intended protection.

### Impact

An unauthenticated attacker on the local network or via a misconfigured reverse proxy could modify tool configurations, alter security policies, or restart the add-on, leading to unauthorized control and potential disruption of the Home Assistant environment.

### Key diff

```
--- a/src/ha_mcp/settings_ui.py
+++ b/src/ha_mcp/settings_ui.py
@@ -3091,8 +3099,8 @@ def register_settings_routes(
         # button is unaffected — its traffic arrives from the Supervisor.
         mcp.custom_route("/", methods=["GET"])(_ingress_only(handlers["root_page"]))
```

### Fix summary

The patch introduces an `_ingress_only` decorator that restricts access to root-mounted add-on routes. It checks the client's transport peer IP address and only allows requests originating from the Home Assistant Supervisor's ingress IP (172.30.32.2). All other direct requests to the root path are now blocked with a 403 Forbidden response.

## GHSA-cwv4-h3j5-w3cf · rama

2026-07-07 · RUST · **LOW** 3.7 · Pattern: `UNSANITIZED_INPUT→XSS`

### Root cause

The vulnerability existed because the `rama` web framework's directory listing feature did not properly escape file names and URI paths when generating HTML output. This allowed an attacker to inject arbitrary HTML or JavaScript code into the directory listing by creating files or directories with malicious names.

### Impact

An attacker could execute arbitrary JavaScript in the context of a user viewing a crafted directory listing, leading to session hijacking, defacement, or redirection to malicious sites.

### Key diff

```

-            Ok(Some(OpenFileOutput::Html(html)))

         }

 }
```

### Fix summary

The patch refactors the HTML generation logic for directory listings into a separate module (`open_file_html.rs`) and ensures that all user-controlled input, specifically file names and URI paths, are properly HTML-escaped before being rendered in the directory listing page. This prevents the injection of malicious scripts.

## GHSA-9f4f-jv96-8766 · open-webui

2026-07-07 · Python · **HIGH** 0.0 · Pattern: `UNSANITIZED_INPUT→XSS`

### Root cause

The application was vulnerable to stored XSS because it directly rendered user-controlled HTML content within markdown tokens without sufficient sanitization. Specifically, certain HTML tokens, including iframes with specific `src` attributes, were rendered using `@html` without being passed through DOMPurify, allowing an attacker to inject malicious scripts.

### Impact

An attacker could inject arbitrary JavaScript code into the application, leading to full account takeover, data exfiltration, or even Remote Code Execution (RCE) by leveraging the application's function calling capabilities.

### Key diff

```

+++ b/src/lib/components/chat/Messages/Markdown/MarkdownInlineTokens.svelte
@@ -23,16 +24,7 @@

 		{unescapeHtml(token.text)}
 	{:else if token.type === 'html'}
-		{@const html = DOMPurify.sanitize(token.text)}
-		{#if html && html.includes('<video')}
-			{@html html}
```

### Fix summary

The patch introduces a new `HTMLToken.svelte` component to centralize the handling of HTML tokens. This component ensures that all HTML content is sanitized using DOMPurify before being rendered, except for specific whitelisted cases like YouTube embeds or internal file iframes, which are handled with strict validation. The `MarkdownInlineTokens.svelte` and `MarkdownTokens.svelte` components now delegate HTML token rendering to this new, safer component.
